Subject: Mobile App Integration: UI to API Gap Report

Hey! I’m currently wiring up the Flutter mobile app UI to your backend APIs. The architecture you’ve built is incredibly solid!

Because the mobile UI designs and models are strictly locked in, we need the API responses to match exactly what each screen requires. After a deep review of every mobile screen against the API payloads provided, I found a few 500 server errors, some HTTP method mismatches in the Settings screens, and missing JSON fields.

Could you please review and update the following?

### 🔴 1. Critical Server Bugs (500 Errors & Formatting)
* **Saved Candidates Crash:** `GET /api/v1/employer/saved-candidates` instantly returns a `500 Internal Server Error`.
* **Resumes Format Issue:** `GET /api/v1/seeker/resumes` is returning `multipart/form-data`. The mobile app needs this to return a standard **JSON array** of the seeker's uploaded resumes.

### ⚙️ 2. Settings & Privacy Mismatches
The mobile app handles Settings slightly differently than the current API structure. Let me know if you can map to the App's flow, or if I should rewrite the App's API calls:
* **Change Email / Phone (Method Mismatch):** The API requires specific endpoints like `PUT /employer/settings/email` and a multi-step OTP flow. However, the Mobile App is currently built to send a generic `PATCH /api/v1/employer/settings` payload (e.g., `{"email": "new@email.com", "current_password": "..."}`) and uses standard `POST /auth/verify-otp` for email. 
* **Change Password (Method & Field Mismatch):** The Mobile App sends `PATCH /employer/settings/password` (instead of `PUT`), and it includes an extra `password_confirmation` field.
* **2FA Toggle:** The Mobile App sends `PATCH /employer/settings/2fa` with the field `"enabled": true` (instead of `"enable": true`).
* **Preferences Mismatch:** The API expects `company_profile_public` (Boolean). The Mobile App UI expects a string mode (e.g., `"public"` or `"private"`). 
* **Missing Feature: Data Export:** The Mobile UI has explicit buttons for **"Export Jobs CSV"** and **"Export Applicants CSV"**. There are currently no API endpoints provided to generate these.

### 🟢 3. Employer UI Screens API Gaps
**Missing Endpoint: Employer Dashboard Screen**
The Employer Home screen requires an endpoint (`GET /api/v1/employer/dashboard`) to fetch aggregate stats for the UI tiles:
* `total_jobs` (Integer)
* `active_jobs` (Integer)
* `total_applicants` (Integer)
* `shortlisted_count` (Integer)
* `recent_jobs` (Array of the 3 most recent jobs)

**Company Setup / Profile Screen (`GET /api/v1/employer/me`)**
The `companies` object requires two read-only counts to display on the company card:
* `active_jobs_count` (Integer)
* `applicants_count` (Integer)

**Job Applicant Card Screen (`GET /api/v1/employer/jobs/:id/applicants`)**
When viewing an applicant, the mobile UI expects to render their *full* profile details. The nested `seeker_profiles` object is missing:
* `phone` (String), `avatar_url` (String), `about` (rich text bio String), `expected_salary` (String), `availability` (String), `cover_letter` (String).
* `skills` (Array of Strings).
* `work_experience` (Array of Objects).
* `education` (Array of Objects).
* `screening_answers` (JSON Object).
* `has_messaged` (Boolean). *(Also needs an endpoint like `POST /employer/applications/:id/message-sent` to toggle this flag).*
